# Meta Information Architecture — Ahd (عهد)

## Goal
Consolidate the project's sprawling status/decision/log files into one source of truth, harden the
multi-agent coordination protocol, and add targeted tool-enforced checks for failure modes that have
already recurred — so the project can scale (more autonomous agent sessions, a human team joining,
production/investor scrutiny, more product surface) without losing track of state or duplicating work.

Product code (`app/features/*`, `app/screens/*`) is out of scope — it's already in good, consistently
tested shape. This is entirely about the *meta* layer: docs, logs, decisions, and agent coordination.

## Why now (evidence gathered 2026-07-01)
- At least 7 status/decision-tracking files exist with no single source of truth: `_meta/STATUS.md`,
  `_meta/overnight-log.md` (547 lines and growing), `_meta/deep-work/ledger/{open-threads,decisions-register,
  sources-ledger}.md`, `docs/DECISIONS-FOR-MARWAN.md`, `docs/specs/{STATUS,build-log}.md`, plus more inside
  the research vault.
- `_meta/deep-work/ledger/open-threads.md` (status: "sealed-v1", updated 2026-06-19) lists 8 P0/P1 items;
  6 of them (OT-SOUL, OT-FSM, OT-CONSENT, OT-PCT, OT-RIBA, OT-STEP0) were already closed the same day by
  `docs/specs/STATUS.md` (items B1–B8), but nothing in open-threads.md marks this — a reader would think
  6 more things are open than actually are.
- The research vault (`docs/research/Amad Obsidian Vault/`) duplicates the same category structure
  (Agent research / Leap / Gauntlet) across two nesting levels for odd- vs. even-numbered agents. A past
  agent's own ledger (`_meta/deep-work/ledger/00_LEDGER.md`) already flagged this: "the round is SPLIT
  across two `10_Deep` trees... provenance is mixed." The same vault also contains its own dead copy of an
  agent-presence system (`AMAD-2026/.agent-presence/`, gitignored, superseded by `_meta/agent-presence/`)
  and an accidentally-committed `.obsidian/` app-config folder.
- `project/` was reused for something unrelated after a restructure: it used to mean "the two builds"
  (`project/ahd-app`, `project/ahd-demo`, since flattened to root `app/`/`demo/`), then a new `project/mcp/`
  was created for the MCP servers. Any doc/memory still saying `project/ahd-app` now describes nothing.
- Live collision observed this session: two sessions (`Claude-D`, `Claude-D-2`) independently ran the same
  review task ~90 seconds apart, and a third (`Claude-B`) independently found and partially fixed some of
  the same issues (deleted a stray root `mcp/` duplicate, patched CLAUDE.md/README) — all three coordinating
  through `_meta/agent-presence/coordination_notes.md` in real time. The presence-file read-then-write race
  (a file changing between READ and WRITE) is a real, reproducible failure mode, not hypothetical.

## Non-goals
- No physical merge of the two historical vault trees (`Amad Obsidian Vault/` root vs. nested `AMAD-2026/`)
  — too risky to merge destructively; pointer/marker docs instead.
- No database, search index, or HTTP transport for any tooling — stays flat-file, consistent with the MCP
  servers' own stated scope (`docs/superpowers/specs/2026-06-30-ahd-mcp-servers-design.md`, "Out of scope").
- No resolving Shariah-gated decisions (D-1, D-3) — those stay for Marwan/counsel.
- No changes to golden functions, the demo tripwire, or any product feature code.
- No editing anything under `project/mcp/**` while Claude-B's hardening-pass claim is active (per
  `_meta/agent-presence/coordination_notes.md`, 2026-07-01T04:26+03:00).

## Part 2 — Agent-coordination hardening

Updates to the agent-awareness protocol as practiced in this project (`_meta/agent-presence/`), grounded in
the real collision observed this session:

1. **Codify the read-then-write race.** The protocol already implies re-reading before writing; make it
   an explicit step: immediately before writing a presence/claim file, re-read it. If it changed since your
   last read, treat that as an **active collision** (register under a suffixed name, do not overwrite),
   regardless of what the timestamp alone would suggest.
2. **Document `_meta/agent-presence/` in CLAUDE.md** (see Part 1) so it's discoverable on the first pass.
3. **Apply the same append-only-forever fix to `coordination_notes.md`** as overnight-log.md: once it
   exceeds ~50 entries, move the oldest half into `_meta/archive/coordination-notes-history/`, keep the
   live file short enough to read in full at session start.

No changes to the claim schema, the 45-minute staleness window, or the newest-first ordering convention —
these are already working as designed.

## Part 3 — Tool-enforced structure checks

Three checks targeting the failure modes with actual evidence (not speculative ones), implemented as a
**new, standalone CommonJS module** — not an edit to any file under `project/mcp/`, so nothing races
Claude-B's in-flight hardening pass there:

- **New file: `tests/structure-check.cjs`** — a sibling of `run-tests.cjs`/`offline-check.cjs`/
  `dom-smoke.cjs` (same self-contained pattern: its own tiny assert framework, `node structure-check.cjs`
  runs and exits 0/1). Exports and runs three checks:
  1. `checkProjectMapFreshness()` — compares `project/mcp/packages/ahd-navigator/src/project-map.json`'s
     `features[]` against the real `app/features/*.js` + `app/screens/*.js` files on disk; fails on any
     drifted/missing entry.
  2. `checkAgentPresenceHealth()` — reads `_meta/agent-presence/*.json`; fails if any `status:"active"`
     entry's `last_heartbeat` is >45 minutes old, or if two active entries claim the same file/task.
  3. `checkSingleStatusFile()` — fails if a status/build-log-shaped file appears outside the two canonical
     locations (`_meta/STATUS.md`, `_meta/overnight-log.md`) or the designated archive.

  **Important — this is NOT auto-discovered.** Unlike `tests/app/`'s suites (auto-discovered by
  `run-app-tests.cjs` via a filename pattern), the core gate (`run-tests.cjs`/`offline-check.cjs`/
  `dom-smoke.cjs`) is three explicitly-named, explicitly-invoked commands — there is no discovery
  mechanism at that level. `structure-check.cjs` must be added as a **fourth explicit command**, and
  CLAUDE.md's documented gate sequence (currently `node run-tests.cjs && node offline-check.cjs && node
  dom-smoke.cjs`) must be updated to include it. TDD: each check gets a fixture proving it correctly flags
  a deliberately-broken case, then proving it passes on the real current state.

**Sequencing constraint:** this part ships as the standalone `tests/structure-check.cjs` script only.
Wiring `checkProjectMapFreshness`/etc. into `ahd-fs` as an actual MCP tool is a **follow-up step**, done
only after Claude-B releases the `project/mcp/**` claim (per `coordination_notes.md`). Post an update
there before starting that follow-up step.
